## 8. 当前不足说明

验收时可以如实说明以下不足：

1. 技术栈偏轻量，适合本地课程演示，不是生产级架构。
2. AI 能力依赖 DeepSeek API，网络或 Key 异常会影响效果。
3. PDF 文本抽取受 PDF 质量影响，扫描版 PDF 效果较弱。
4. 当前测试以单元测试为主，接口测试和端到端测试还需补齐。
5. 真实论文检索能力有限，后续应接入开放论文 API。
6. 课堂汇报目前是结构化文本，后续可增强为 Markdown、PPT 或演示页面。

这些不足不会否定项目价值，反而能体现对项目边界和后续迭代的清晰认识。

## 11. 结论

当前 ScholarFlow 已经具备课程验收的基础形态：有代码、有功能、有文档、有测试、有路线图、有 GitHub 仓库。若要进一步提高项目质量，应优先围绕“使用效果”继续开发，而不是急于升级技术栈。

最合理的下一步是：

```text
错题本 -> 能力画像 -> 课堂汇报导出 -> 接口测试 -> 端到端测试
```

这条路线能最大化提升项目实用性和验收说服力。
