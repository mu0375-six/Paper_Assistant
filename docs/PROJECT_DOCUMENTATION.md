# ScholarFlow 项目文档

## 1. 项目概述

### 1.1 项目名称

ScholarFlow / Paper Assistant。

### 1.2 项目定位

ScholarFlow 是一个面向学生和初级研究者的 AI 辅助论文阅读与学习训练系统。项目目标不是简单地让 AI 代替用户总结论文，而是帮助用户完成一套可复用的学术阅读流程：

```text
研究方向检索 -> 候选论文筛选 -> PDF 阅读 -> AI 精读 -> 闯关问答纠错 -> 笔记沉淀 -> 课堂汇报
```

### 1.3 当前项目状态评估

当前版本已经具备演示型原型能力，但距离“成熟可验收项目”仍有差距。

| 维度 | 当前状态 | 主要问题 |
| --- | --- | --- |
| 技术栈 | Flask + 原生 HTML/CSS/JS + JSON 文件存储 | 适合课程演示，但工程化不足，前端状态复杂，后端服务文件过大 |
| 实用性 | 支持上传 PDF、AI 精读、闯关、笔记、示例教程 | 工作流还不够闭环，检索结果和真实论文数据库连接有限 |
| 设计完整性 | 已有页面和接口，但缺少正式设计说明 | 模块边界、数据流、异常流程、验收标准不够明确 |
| 测试 | 已有 6 个单元测试 | 测试覆盖较低，缺少接口测试、端到端测试、性能测试和安全测试 |
| 可维护性 | 可运行，但部分文件承担过多职责 | `scholar_service.py` 与 `web/app.js` 文件过大，后续修改风险高 |

## 2. 项目背景与问题分析

### 2.1 用户痛点

学生阅读论文时常见问题包括：

- 不知道第一步该看哪里。
- 容易把“摘要复述”误认为“读懂论文”。
- 不会区分研究问题、方法流程、实验结果和局限性。
- 看完论文后没有形成笔记和汇报材料。
- 使用通用 AI 工具时，容易得到一段总结，但缺少学习训练过程。

### 2.2 项目价值

ScholarFlow 的价值应从“AI 生成答案”转向“AI 训练用户读懂论文”。因此项目的核心卖点不是摘要生成，而是：

- 提供新手阅读路线。
- 引导用户分步骤理解论文。
- 通过闯关问答暴露理解缺口。
- 将 AI 纠错结果沉淀为笔记和课堂汇报。

## 3. 需求分析

### 3.1 功能性需求

| 编号 | 功能 | 说明 | 优先级 |
| --- | --- | --- | --- |
| FR-01 | 新手教程 | 提供 6 步论文阅读路线和按钮说明 | 高 |
| FR-02 | 示例论文 | 内置示例论文，便于首次体验 | 高 |
| FR-03 | PDF 导入 | 支持上传 PDF 并抽取文本 | 高 |
| FR-04 | PDF 原貌阅读 | 保留 PDF 原始排版、公式和图表 | 高 |
| FR-05 | 文本层选区 | 支持对抽取文本进行解释、翻译、提问和保存笔记 | 中 |
| FR-06 | AI 精读 | 生成摘要、关键词、研究问题、方法和推荐追问 | 高 |
| FR-07 | AI 闯关 | 通过 5 个问题训练论文理解能力 | 高 |
| FR-08 | 评分纠错 | 对用户答案给出分数、反馈和参考修正 | 高 |
| FR-09 | 课堂汇报 | 根据闯关结果生成汇报结构 | 中 |
| FR-10 | 笔记写作 | 保存阅读笔记、AI 回答和写作草稿 | 中 |
| FR-11 | 研究检索 | 根据研究方向生成候选论文和筛选建议 | 中 |

### 3.2 非功能性需求

| 编号 | 需求 | 说明 |
| --- | --- | --- |
| NFR-01 | 可用性 | 新用户能通过教程完成第一次阅读流程 |
| NFR-02 | 可靠性 | API 异常、PDF 抽取失败时页面不能崩溃 |
| NFR-03 | 可测试性 | 核心业务逻辑应有单元测试和接口测试 |
| NFR-04 | 可维护性 | 后端服务和前端控制器需要逐步拆分 |
| NFR-05 | 数据安全 | API Key 不写入代码，运行数据不提交 GitHub |
| NFR-06 | 响应时间 | 非 AI 接口响应应控制在 1 秒内，AI 接口应有明确等待提示 |

## 4. 当前系统设计

### 4.1 技术架构

```text
浏览器页面
  |
  | HTTP / JSON
  v
Flask 路由层 app/main.py
  |
  v
业务服务层 app/scholar_service.py
  |
  |-- DeepSeekClient: 调用 DeepSeek API
  |-- JsonStore: 读写本地 JSON 数据
  |-- pypdf: PDF 文本和图片抽取
```

### 4.2 前端页面

| 页面 | 文件 | 说明 |
| --- | --- | --- |
| 首页 | `web/home.html` | 项目入口、最近文献、最近检索 |
| 研究检索 | `web/discovery.html` | 输入研究方向，展示候选论文 |
| 阅读台 | `web/reader.html` | PDF 阅读、AI 精读、闯关、问答、笔记 |
| 笔记写作 | `web/workspace.html` | 阅读笔记、研究计划、写作草稿 |

### 4.3 后端核心模块

| 模块 | 文件 | 职责 |
| --- | --- | --- |
| Flask 应用入口 | `app/__init__.py` | 创建 Flask app |
| 路由层 | `app/main.py` | 注册页面路由和 API 路由 |
| 业务服务 | `app/scholar_service.py` | 文献、AI 分析、闯关、笔记、写作逻辑 |
| LLM 客户端 | `app/deepseek_client.py` | DeepSeek API 请求封装 |
| 数据存储 | `app/storage.py` | JSON 文件读写 |

### 4.4 数据模型

#### Paper

| 字段 | 说明 |
| --- | --- |
| `id` | 文献唯一 ID |
| `title` | 标题 |
| `authors` | 作者 |
| `year` | 年份 |
| `source` | 来源 |
| `abstract` | 摘要 |
| `content` | 抽取正文 |
| `pages` | 按页抽取文本 |
| `figures` | 抽取图片信息 |
| `downloaded_pdf_url` | 本地 PDF 地址 |
| `analysis` | AI 精读结果 |
| `tutorial_sample` | 是否为内置示例论文 |

#### ChallengeRun

| 字段 | 说明 |
| --- | --- |
| `paper_id` | 关联论文 |
| `stages` | 5 个闯关阶段 |
| `attempts` | 每关回答历史 |
| `score` | AI 评分 |
| `feedback` | AI 反馈 |
| `correction` | 参考修正 |
| `evidence_hint` | 证据提示 |

#### Note

| 字段 | 说明 |
| --- | --- |
| `paper_id` | 关联论文 |
| `title` | 笔记标题 |
| `content` | 笔记内容 |
| `kind` | 来源类型，如 selection、assistant |
| `source_text` | 原始选区或 AI 回答来源 |
| `page_label` | 页码 |
| `ai_prompt` | AI 问答提示词 |

## 5. 核心业务流程

### 5.1 新手第一次使用流程

```text
打开阅读台
  -> 查看新手教程
  -> 导入示例论文
  -> 点击 AI 精读
  -> 查看摘要和关键词
  -> 进入闯关
  -> 回答研究问题
  -> 查看评分纠错
  -> 保存笔记
  -> 生成课堂汇报
```

### 5.2 真实论文阅读流程

```text
上传 PDF
  -> 原文阅读
  -> 打开文本层选区
  -> 选中关键句
  -> 向 AI 提问或保存笔记
  -> AI 精读
  -> 闯关评分
  -> 笔记写作
```

### 5.3 AI 闯关流程

| 关卡 | 训练能力 | 用户任务 |
| --- | --- | --- |
| 第 1 关 | 问题识别 | 说明论文想解决什么问题 |
| 第 2 关 | 方法理解 | 按输入、步骤、输出拆解方法 |
| 第 3 关 | 证据判断 | 解释图表证明了什么和不能证明什么 |
| 第 4 关 | 批判思维 | 指出局限和适用边界 |
| 第 5 关 | 研究延展 | 提出后续研究问题 |

## 6. 当前不足与改进方向

### 6.1 改进优先级说明

当前阶段暂不升级技术栈。原因是项目首先需要达到课程验收要求，而不是过早迁移框架。现阶段更重要的是：

- 提升使用效果，让系统真正帮助用户完成论文阅读。
- 补齐项目必须项，包括文档、测试、验收清单和演示脚本。
- 明确功能递增路线，避免后续开发变成零散加功能。

技术栈升级可放到后续工程化版本中考虑。

### 6.2 使用效果改进

| 问题 | 改进方向 |
| --- | --- |
| AI 反馈有时不稳定 | 固定 JSON Schema，增加输出校验和重试 |
| PDF 文本抽取不稳定 | 引入更稳健的 PDF 解析策略，保留原文 PDF 作为主阅读视图 |
| 闯关不够像学习系统 | 增加错题本、能力雷达图、薄弱点追踪 |
| 课堂汇报还不够完整 | 生成可展示页面或 Markdown/PPT 提纲 |
| 检索不够真实 | 接入开放论文 API，如 arXiv、Semantic Scholar 或 Crossref |

### 6.3 使用效果优先开发项

| 优先级 | 功能 | 说明 | 预期效果 |
| --- | --- | --- | --- |
| 1 | 错题本 | 保存低分答案、AI 修正和证据提示 | 让 AI 纠错沉淀为学习记录 |
| 2 | 能力画像 | 根据 5 个闯关维度计算能力分布 | 让学习效果可视化 |
| 3 | 阅读进度 | 标记教程、精读、闯关、笔记、汇报完成状态 | 让用户知道下一步做什么 |
| 4 | 课堂汇报导出 | 支持复制 Markdown 或生成独立汇报页 | 提高课堂展示价值 |
| 5 | 候选论文对比 | 对比多篇论文的研究问题、方法和局限 | 提高检索后的筛选效率 |

## 7. 验收标准

### 7.1 最小可验收标准

项目达到以下条件，可以作为课程项目验收：

- 能在本地通过 README 步骤启动。
- 首页、研究检索、阅读台、笔记写作页面均可访问。
- 能导入示例论文。
- 能上传 PDF 并打开阅读台。
- 能完成 AI 精读或 fallback 精读。
- 能完成至少 1 个闯关问题并得到评分反馈。
- 能生成课堂汇报。
- 能保存笔记。
- 自动化测试全部通过。
- 项目文档和测试文档完整。

### 7.2 推荐验收演示脚本

1. 展示首页，说明项目定位。
2. 打开阅读台，展示新手教程。
3. 导入示例论文。
4. 点击 AI 精读，展示摘要和推荐问题。
5. 进入闯关，提交第 1 关答案。
6. 展示 AI 评分、纠错和重新作答。
7. 生成课堂汇报。
8. 保存一条笔记，打开笔记写作页查看。
9. 运行测试命令，展示测试通过。

## 9. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| DeepSeek API 不可用 | AI 功能失败 | 提供 fallback 结果和错误提示 |
| PDF 抽取失败 | 文本层不可用 | 保留原始 PDF 阅读 |
| LLM 输出格式错误 | 页面展示异常 | 增加 JSON 校验和默认值 |
| 数据文件损坏 | 本地数据丢失 | 增加备份和初始化逻辑 |
| 项目范围过大 | 无法按时完成 | 以课程验收版为优先目标 |

## 10. 总结

ScholarFlow 当前不是一个成熟产品，但已经具备成为课程验收项目的基础。后续重点不应继续堆叠零散功能，而应围绕“论文阅读训练闭环”补齐三类内容：

- 工程层面：模块化、数据存储、异常处理。
- 教学层面：教程、闯关、错题本、能力反馈。
- 验收层面：项目文档、测试文档、测试用例和演示脚本。
