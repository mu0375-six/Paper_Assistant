# ScholarFlow 功能递增开发路线图

## 1. 路线图目标

本路线图用于指导 ScholarFlow 从“可演示原型”逐步提升到“课程可验收项目”。当前阶段暂不升级技术栈，仍然保留：

- Flask 后端
- 原生 HTML / CSS / JavaScript
- JSON 文件本地存储
- DeepSeek API

路线图重点放在三个方向：

- 使用效果改进：让用户真的能按步骤读论文，而不是只看到功能堆叠。
- 项目必须项补齐：补足文档、测试、演示脚本、验收材料。
- 功能递增开发：每一阶段都有明确可见的成果和验收标准。

## 2. 总体开发原则

### 2.1 先闭环，再扩展

优先打通完整学习闭环：

```text
导入论文 -> 阅读引导 -> AI 精读 -> 闯关纠错 -> 笔记沉淀 -> 课堂汇报
```

只有这个闭环稳定后，再考虑增加更多检索源、图表分析或复杂写作功能。

### 2.2 先可验收，再高级化

课程验收看重的是项目完整性：

- 有明确需求。
- 有设计过程。
- 有核心功能。
- 有测试用例。
- 有测试结果。
- 有项目文档。
- 有演示材料。

因此开发优先级应高于单纯“功能炫酷”。

### 2.3 每一阶段都能展示

每个阶段都应产出可展示成果，而不是只做内部改动。

## 3. 当前版本基线

当前版本已经完成：

| 模块 | 当前能力 |
| --- | --- |
| 首页 | 展示项目入口、最近文献和最近检索 |
| 研究检索 | 输入研究方向，返回候选论文和筛选建议 |
| 阅读台 | 支持 PDF 阅读、文本层选区、AI 问答 |
| AI 精读 | 生成摘要、关键词、研究问题和推荐追问 |
| 闯关问答 | 5 个论文精读关卡，支持评分和纠错 |
| 课堂汇报 | 根据闯关结果生成汇报结构 |
| 笔记写作 | 保存笔记、生成研究计划和写作草稿 |
| 新手教程 | 内置 6 步阅读路线和示例论文 |
| 自动化测试 | 6 个单元测试，覆盖示例论文、闯关、笔记来源 |

当前主要不足：

- 闯关结果没有形成长期学习档案。
- 缺少错题本、能力画像和阅读进度。
- 课堂汇报仍偏文本，不够像最终展示材料。
- 自动化测试覆盖不足，缺少接口测试和端到端测试。
- 文档体系刚刚补齐，还需要和开发进度持续同步。

## 4. 阶段一：验收基础补齐

### 4.1 目标

让项目达到“课程验收基础完整”的状态。

### 4.2 功能范围

| 编号 | 功能 | 说明 |
| --- | --- | --- |
| P1-01 | 文档体系补齐 | 项目文档、测试文档、路线图、验收清单 |
| P1-02 | README 完善 | 增加文档入口、测试命令、演示流程 |
| P1-03 | 启动说明明确 | 明确 API Key、模型、依赖安装和运行命令 |
| P1-04 | 演示脚本 | 提供课堂展示步骤 |
| P1-05 | 测试命令统一 | README、测试文档、CI 使用一致测试命令 |
| P1-06 | GitHub Actions | 自动运行 Python 测试和 JS 语法检查 |

### 4.3 交付物

- `docs/PROJECT_DOCUMENTATION.md`
- `docs/TEST_DOCUMENTATION.md`
- `docs/ROADMAP.md`
- `docs/ACCEPTANCE_CHECKLIST.md`
- 更新后的 `README.md`
- 可运行的 GitHub Actions CI

### 4.4 验收标准

- 文档能解释项目是什么、怎么设计、怎么测试、怎么演示。
- 执行测试命令后全部通过。
- GitHub 仓库中不包含真实 API Key、运行数据和缓存文件。
- 老师可以根据文档复现项目运行过程。

### 4.5 建议时间

1-2 天。

## 5. 阶段二：使用效果增强

### 5.1 目标

让项目从“能点功能”提升为“真的能帮助读论文”。

### 5.2 功能范围

| 编号 | 功能 | 说明 | 价值 |
| --- | --- | --- | --- |
| P2-01 | 阅读进度条 | 标记用户完成了教程、AI 精读、闯关、笔记、汇报哪些步骤 | 让用户知道下一步做什么 |
| P2-02 | 当前任务提示 | 在阅读台顶部显示“建议下一步” | 降低使用迷茫 |
| P2-03 | 错题本 | 保存低分闯关答案、AI 修正、证据提示 | 让 AI 纠错变成学习记录 |
| P2-04 | 能力画像 | 根据 5 个闯关维度显示能力得分 | 让学习效果可视化 |
| P2-05 | 阅读完成状态 | 每篇论文显示“未读 / 精读中 / 可汇报” | 增强文献库管理能力 |

### 5.3 推荐实现顺序

1. 先增加阅读进度状态。
2. 再增加错题本。
3. 然后增加能力画像。
4. 最后在文献列表中显示阅读状态。

### 5.4 数据设计建议

可以在现有 JSON 数据基础上扩展，不需要升级数据库。

```json
{
  "paper_id": "paper-1",
  "completed_steps": ["tutorial", "analyze", "challenge_problem", "note"],
  "weaknesses": ["method", "evidence"],
  "last_action": "challenge",
  "status": "reading"
}
```

### 5.5 验收标准

- 用户进入阅读台后能看到下一步建议。
- 低分答案会进入错题本。
- 闯关结果能生成 5 项能力分布。
- 文献库能显示每篇论文阅读状态。

### 5.6 建议时间

3-5 天。

## 6. 阶段三：课堂汇报增强

### 6.1 目标

让“课堂汇报”成为项目的亮点输出，而不是简单文本摘要。

### 6.2 功能范围

| 编号 | 功能 | 说明 |
| --- | --- | --- |
| P3-01 | 汇报封面 | 自动生成论文题目、作者、年份、汇报人 |
| P3-02 | 汇报结构 | 研究背景、问题、方法、证据、局限、个人理解 |
| P3-03 | 闯关纠错记录 | 展示用户原答案、AI 纠错和最终答案 |
| P3-04 | 一键复制 Markdown | 将汇报内容复制为 Markdown |
| P3-05 | 汇报演示页 | 提供独立 `/presentation` 或阅读台内演示模式 |

### 6.3 推荐输出结构

```text
1. 论文基本信息
2. 为什么读这篇论文
3. 研究问题
4. 方法流程
5. 关键证据
6. 局限与质疑
7. 我的理解与后续问题
8. AI 闯关纠错记录
```

### 6.4 验收标准

- 至少能生成一份完整汇报稿。
- 汇报稿中包含闯关结果。
- 用户可以复制汇报内容用于课堂展示。
- 汇报结构比普通摘要更清晰。

### 6.5 建议时间

2-3 天。

## 7. 阶段四：测试体系补强

### 7.1 目标

让项目测试从“少量单元测试”提升到“覆盖核心流程的测试体系”。

### 7.2 功能范围

| 编号 | 测试类型 | 说明 |
| --- | --- | --- |
| P4-01 | API 测试 | 使用 Flask test client 测接口 |
| P4-02 | 异常测试 | 测空输入、错误 paper_id、空答案、API 未配置 |
| P4-03 | PDF 测试 | 使用小型测试 PDF 验证上传 |
| P4-04 | 端到端测试 | 用 Playwright 测新手教程、示例论文、闯关 |
| P4-05 | 安全测试脚本 | 搜索 API Key、检查 data 是否被提交 |

### 7.3 推荐新增测试文件

```text
tests/test_api_routes.py
tests/test_error_handling.py
tests/test_pdf_import.py
tests/test_security_checks.py
tests/e2e/test_reader_flow.py
```

### 7.4 验收标准

- 单元测试覆盖核心业务逻辑。
- 接口测试覆盖主要 API。
- 端到端测试覆盖一次完整阅读流程。
- CI 自动运行全部基础测试。
- 测试文档与实际测试文件对应。

### 7.5 建议时间

3-4 天。

## 8. 阶段五：检索与真实可用性增强

### 8.1 目标

让研究检索不再只是“演示候选论文”，而是更接近真实学习场景。

### 8.2 功能范围

| 编号 | 功能 | 说明 |
| --- | --- | --- |
| P5-01 | 开放论文 API | 优先考虑 arXiv、Crossref、Semantic Scholar |
| P5-02 | 检索结果筛选 | 按年份、来源、开放 PDF、关键词筛选 |
| P5-03 | 检索历史恢复 | 点击历史可恢复候选论文 |
| P5-04 | 候选论文对比 | 对比研究问题、方法、优缺点 |
| P5-05 | 加入文献库 | 从候选论文一键加入阅读台 |

### 8.3 验收标准

- 输入研究方向后能返回真实来源的候选论文。
- 每条候选论文有来源信息。
- 能把候选论文加入文献库。
- 检索历史可恢复结果。

### 8.4 建议时间

4-7 天。

## 9. 阶段六：最终验收整理

### 9.1 目标

整理所有材料，形成可提交、可讲解、可复现的课程项目。

### 9.2 交付物

| 类型 | 文件或内容 |
| --- | --- |
| 代码 | GitHub 仓库 |
| 项目说明 | README |
| 项目文档 | `docs/PROJECT_DOCUMENTATION.md` |
| 测试文档 | `docs/TEST_DOCUMENTATION.md` |
| 开发路线图 | `docs/ROADMAP.md` |
| 验收清单 | `docs/ACCEPTANCE_CHECKLIST.md` |
| 演示脚本 | README 或单独演示文档 |
| 测试结果 | 终端测试截图或 CI 记录 |

### 9.3 验收标准

- 老师能根据文档理解项目设计。
- 项目能在本地启动。
- 核心流程能现场演示。
- 测试用例能运行。
- GitHub 仓库结构清晰。
- 项目说明不夸大能力，能解释当前不足和后续规划。

### 9.4 建议时间

1-2 天。

## 10. 推荐开发时间线

| 时间 | 阶段 | 主要任务 | 输出 |
| --- | --- | --- | --- |
| 第 1 天 | 阶段一 | 补齐文档、README、验收清单 | 文档体系完整 |
| 第 2-3 天 | 阶段二 | 阅读进度、错题本、能力画像 | 使用效果增强 |
| 第 4-5 天 | 阶段三 | 课堂汇报增强、Markdown 导出 | 展示效果增强 |
| 第 6-8 天 | 阶段四 | API 测试、异常测试、E2E 测试 | 测试体系增强 |
| 第 9-12 天 | 阶段五 | 真实检索源、结果筛选、文献对比 | 实用性增强 |
| 第 13 天 | 阶段六 | 整理文档、演示脚本、测试结果 | 最终验收材料 |

## 11. 优先级建议

如果时间有限，优先做：

1. 错题本。
2. 能力画像。
3. 课堂汇报 Markdown 导出。
4. API 测试。
5. 端到端测试。

如果时间充足，再做：

1. 真实论文 API 检索。
2. 候选论文对比。
3. 阅读进度长期记录。
4. 演示模式页面。

## 12. 总结

短期内不要急着升级技术栈。当前更重要的是把 ScholarFlow 从“功能堆叠型原型”改造成“学习闭环型项目”。

最有价值的开发方向是：

- 让用户知道下一步该做什么。
- 让 AI 反馈沉淀成错题和能力画像。
- 让课堂汇报成为可复制、可展示的最终产物。
- 让测试和文档能支撑课程验收。
