# ScholarFlow 答辩讲解文档

> 本文档专为课程答辩设计，涵盖技术栈、架构设计、开发流程、测试技术四大板块，可按需选取段落作为讲解话术。

---

## 一、项目简介（30 秒版）

ScholarFlow 是一个 AI 辅助论文阅读训练系统。和普通"AI 帮你读论文"不同，它不替代阅读，而是把论文阅读拆成五关闯关，让学生在回答问题、接受纠错的过程中真正读懂论文。整个工作流覆盖从检索论文、精读分析、闯关训练到课堂汇报的完整闭环。

## 二、技术栈

### 2.1 整体技术选型

| 层级 | 技术 | 选型理由 |
|---|---|---|
| 后端框架 | **FastAPI** | 自动生成 Swagger API 文档、Pydantic 请求校验、原生异步支持、类型标注 |
| 请求校验 | **Pydantic v2** | 用 `RequiredStr` 类型自动拒绝空字符串和纯空格，422 返回详细校验错误，替代手写 if 判断 |
| AI 服务 | **DeepSeek API** | 国产大模型，成本低、中文能力强，支持 JSON 模式输出 |
| 学术检索 | **arXiv API** | 免费、无需 API Key、支持 XML 格式返回，实现真实论文检索 |
| PDF 解析 | **pypdf** | 轻量级，支持文本抽取、图片提取、元数据读取 |
| 数据存储 | **JSON 文件** | 零配置，本地即可运行，适合课程演示 |
| 环境管理 | **python-dotenv** | .env 文件自动加载，无需每次手动设置 API Key |
| Web 服务器 | **uvicorn** | ASGI 标准生产级服务器，支持热重载 |
| 前端 | **HTML / CSS / JavaScript** | 原生实现，无框架依赖，降低部署复杂度 |

### 2.2 为什么选 FastAPI 而不是 Flask

| 对比项 | Flask | FastAPI |
|---|---|---|
| API 文档 | 需要额外安装 flasgger | **自动生成 Swagger UI**，/docs 直接查看和测试 |
| 请求校验 | 手写 `payload.get() or "").strip()` | **Pydantic 模型自动校验**，空值/类型错误自动返回 422 |
| 错误处理 | 手写 try/except | **HTTPException + 422 自动响应** |
| 异步支持 | 不支持 | **原生 async**，未来可扩展为异步处理 |
| 类型标注 | 无 | **全链路类型提示**，IDE 补全友好 |

### 2.3 为什么选 Pydantic 做请求校验

传统写法（每个路由都要手写）：

```python
topic = (payload.get("topic") or "").strip()
if not topic:
    return jsonify({"error": "topic is required"}), 400
```

Pydantic 写法（声明即校验）：

```python
class SearchRequest(BaseModel):
    topic: RequiredStr = Field(description="检索主题关键词")
```

好处：空字符串、纯空格、缺失字段全部自动拦截，返回 422 + 详细错误信息，路由函数只写业务逻辑。

## 三、架构设计

### 3.1 分层架构

```text
┌─────────────────────────────────────────────┐
│            前端 (HTML/CSS/JS)                │
├─────────────────────────────────────────────┤
│         FastAPI 路由层 (main.py)             │
│     Pydantic 校验 → HTTPException 错误处理     │
├─────────────────────────────────────────────┤
│        ScholarService 业务服务层              │
│  ┌──────────┬──────────┬──────────┬────────┐ │
│  │Discovery │  Paper   │Challenge │Writing │ │
│  │  Mixin   │  Mixin   │  Mixin   │ Mixin  │ │
│  └──────────┴──────────┴──────────┴────────┘ │
├─────────────────────────────────────────────┤
│  DeepSeekClient  │  JsonStore  │  arXiv API  │
└─────────────────────────────────────────────┘
```

### 3.2 Mixin 组合模式

原来 `scholar_service.py` 有 1400+ 行，所有逻辑挤在一个文件里。重构后采用 **Mixin 组合模式**：

```python
class ScholarService(PaperMixin, DiscoveryMixin, ChallengeMixin, WritingMixin):
    def __init__(self, store, llm_client):
        self.store = store
        self.llm_client = llm_client
```

四个 Mixin 各自独立，共享 `self.store` 和 `self.llm_client`，对外接口完全不变。这样做的好处：

- **单一职责**：每个模块 200-300 行，改一处不影响其他
- **可测试**：可以单独测某个 Mixin 的逻辑
- **可扩展**：新增功能只需加一个 Mixin

### 3.3 四个服务模块

#### DiscoveryMixin — 检索与发现

系统入口，帮用户找到相关论文。核心亮点是**中文→英文查询翻译**：用户输入"多模态智能体的研究与演进"，系统自动转成 `multimodal agent evolution advancement` 去查 arXiv，内置 60+ 中英学术关键词映射。arXiv 请求失败时降级到内置 8 篇经典论文目录。

#### PaperMixin — 论文管理

最大的模块，覆盖论文全生命周期。核心亮点是**AI 降级策略**：DeepSeek API 不可用时，所有 AI 功能（精读、翻译、问答）自动走 fallback 路径，用户不会看到 500 错误，而是得到基于规则生成的简化结果。PDF 处理自动猜测标题、作者、年份、摘要，并抽取图片。

#### ChallengeMixin — 闯关训练

项目的核心特色。五关闯关分别训练问题识别、方法理解、证据判断、批判思维、研究延展。AI 评分有**分数归一化**机制：模型有时返回 0-1 的小数（如 score=1, level=pass），系统自动转为 80 分以上，避免显示不合理分数。

#### WritingMixin — 写作与研究计划

把阅读成果转化为写作产出。自动把论文分析结果和笔记格式化为 LLM 提示词，生成周计划、大纲和草稿。Fallback 时基于模板生成四段式结构。

### 3.4 AI 降级策略

所有依赖 LLM 的功能都实现了 fallback 降级：

| API 可用 | API 不可用 |
|---|---|
| DeepSeek 返回结构化精读分析 | 基于摘要和标题生成简易分析 |
| DeepSeek 按 0-100 评分 + 中文反馈 | 基于长度 + 关键词匹配规则评分 |
| DeepSeek 双语翻译 | 直接返回原文 |
| DeepSeek 基于上下文回答 | 基于已有分析生成引导式回答 |

这保证了**系统在任何情况下都不崩溃、不返回 500**。

## 四、开发流程

### 4.1 开发阶段

```text
阶段 1：基础搭建
  → Flask 路由 + JSON 存储 + 前端页面
  → 基础 CRUD（论文、笔记）

阶段 2：AI 功能集成
  → 接入 DeepSeek API
  → 实现 AI 精读、翻译、问答
  → 实现闯关评分

阶段 3：功能完善
  → 接入 arXiv API，实现真实论文检索
  → 中文→英文查询翻译
  → 课堂汇报生成
  → python-dotenv 自动加载配置

阶段 4：架构优化
  → scholar_service.py 1400 行拆分为 4 个 Mixin + 辅助模块
  → Flask → FastAPI 迁移
  → 引入 Pydantic 请求校验
  → 统一错误处理（ValueError → 404，校验失败 → 422）

阶段 5：测试与文档
  → 补充 API 接口测试（44 个）
  → 补充服务层边界测试（40 个）
  → 补充手动测试用例（36 个）
  → 更新项目文档和技术栈文档
```

### 4.2 Git 工作流

- 主分支 `main` 为稳定版本
- 开发在 `master` 分支进行
- `.gitignore` 排除 `__pycache__`、`data/`、`.env` 等文件
- API Key 通过 `.env` 管理，不提交到 GitHub

### 5.2 测试方法覆盖

按照软件测试课程的标准分类：

| 测试方法 | 本项目应用 |
|---|---|
| **黑盒测试** | API 接口测试：只关注输入输出，不关心内部实现 |

| **性能测试** | 手动测试记录：非 AI 接口 < 1s，AI 接口 < 30s |

### 5.3 FakeLLM 测试模式

测试时不调用真实 API，使用 FakeLLM 替身：

```python
class FakeLLM:
    model = "test-model"
    def is_configured(self):
        return False  # 使服务走 fallback 路径
```

还有 `FakeLowScaleLLM`（模拟模型返回 0-1 小数）和 `FakeFailLLM`（模拟模型调用异常），确保所有降级路径都被测试覆盖。

### 5.4 Pydantic 校验测试

迁移到 FastAPI 后，原来的手动 400 校验变成了 Pydantic 自动 422 校验：

| 场景 | Flask 时期 | FastAPI 时期 |
|---|---|---|
| 空字符串 title | 手写 if 返回 400 | Pydantic `min_length=1` 自动返回 422 |
| 纯空格 title | 手写 strip() 判断返回 400 | Pydantic `strip_whitespace=True` 自动返回 422 |
| 缺失必填字段 | 手写 if 返回 400 | Pydantic 自动返回 422 + 缺失字段名 |
| 类型错误 | 不校验 | Pydantic 自动返回 422 + 类型提示 |

### 5.5 重构回归测试

项目经历两次重大重构，每次都通过全部测试验证无回归：

| 重构 | 测试数量 | 结果 |
|---|---|---|
| scholar_service.py 拆分为 4 个 Mixin | 93 个 | 全部通过 |
| Flask → FastAPI 迁移 | 93 个 | 全部通过 |

这证明了**服务层与框架解耦**的架构设计是有效的：路由层可以自由更换，业务逻辑不受影响。

### 5.6 Swagger UI 在线测试

FastAPI 自动生成 `/docs` 页面，答辩时可以直接在浏览器中：

1. 打开 `http://localhost:5000/docs`
2. 选择任意 API 端点
3. 点击 "Try it out"
4. 填写参数，点击 "Execute"
5. 实时查看请求和响应

这是 Flask 无法自动提供的。

## 七、答辩话术参考

### 开场（1 分钟）

> 我的项目叫 ScholarFlow，是一个 AI 辅助论文阅读训练系统。和普通"AI 帮你读论文"不同，它不替代阅读，而是把论文阅读拆成五关闯关——问题识别、方法理解、证据判断、局限分析、延伸问题——让学生在回答问题、接受 AI 纠错的过程中真正读懂论文。

### 技术栈（1 分钟）

> 后端我用的是 FastAPI 框架，配合 Pydantic 做请求自动校验。FastAPI 的好处是自动生成了 Swagger API 文档，26 个端点可以在 /docs 页面直接测试。请求校验不再需要手写 if 判断，Pydantic 声明一个 RequiredStr 类型，空字符串和纯空格就自动返回 422 错误。AI 服务用的是 DeepSeek API，学术检索接入了 arXiv API，支持中文搜索词自动翻译成英文去查。

### 架构设计（1 分钟）

> 后端服务层采用 Mixin 组合模式，原来 1400 行的单文件拆成了检索、论文、闯关、写作四个模块，每个 200-300 行，职责单一。所有依赖 AI 的功能都实现了 fallback 降级策略，API 不可用时自动走规则生成的简化结果，系统不会崩溃也不会返回 500。这个设计在后面框架迁移时也验证了有效性——从 Flask 迁移到 FastAPI，服务层一行代码都没改。

### 测试（1 分钟）

> 测试方面，我一共写了 134 个测试用例。93 个自动化测试覆盖了 API 接口、服务层边界和业务逻辑，36 个手动测试用例覆盖了页面交互和答辩演示流程。测试用了 FakeLLM 模式，不依赖真实 API，所以测试在任何环境都能跑。项目经历了两次重大重构——单文件拆分和框架迁移——每次重构后全部测试都通过，证明架构是解耦的。

### 演示（2-3 分钟）

> 下面我演示一下核心流程。打开首页，进入阅读台，导入示例论文，点击 AI 精读可以看到结构化分析结果。然后进入闯关，回答第一关"这篇论文想解决什么问题"，AI 会给评分和纠错。最后生成课堂汇报。另外在 /docs 页面可以看到自动生成的 API 文档，每个接口都能直接在线测试。
